Spring Boot Crash Course

Learn Spring Boot first, step by step, using your Booking Service as the example project.

01
Goal

Understand what a Spring Boot app is

A Spring Boot app is a Java program that can start a web server, receive HTTP requests, run your business logic, and return responses. Your project is a REST API, so its job is not to show pages. Its job is to accept data and return JSON.

  • In your app: the booking service starts on port 8081.
  • Main file: BookingServiceApplication.java.
  • Say: "Spring Boot starts my service and prepares the web, validation, and database layers."
02
Startup

Learn the main class

The class with @SpringBootApplication is the starting point. When the main method runs, Spring Boot creates the application context. The application context is like a container that holds all Spring-managed objects. 

public static void main(String[] args) {
    SpringApplication.run(BookingServiceApplication.class, args);
}
  • What happens: Spring scans your project, creates beans, loads config, starts server.
  • Why it matters: You do not manually create every object with new.
03
Beans

Understand beans and dependency injection

A bean is an object created and managed by Spring. Dependency injection means Spring gives one class the objects it needs. This keeps your code cleaner and easier to test. 

public BookingService(
        BookingRepository bookingRepository,
        TicketingClient ticketingClient,
        PaymentClient paymentClient) {
    this.bookingRepository = bookingRepository;
    this.ticketingClient = ticketingClient;
    this.paymentClient = paymentClient;
}
  • In your app: BookingService needs a repository and two clients.
  • Spring's job: create those objects and pass them into the constructor.
  • Say: "I use constructor injection so dependencies are explicit and testable."
04
HTTP

Understand controllers

A controller is the class that receives web requests. It should be thin. That means it should not contain the main business logic. It should receive the request, validate it, and call the service. 

@PostMapping("/bookings")
@ResponseStatus(HttpStatus.CREATED)
public BookingResponse createBooking(@Valid @RequestBody CreateBookingRequest request) {
    return bookingService.createBooking(request);
}
  • Route: POST /bookings.
  • Input: JSON request body.
  • Output: JSON booking response.
05
DTOs

Understand request and response DTOs

DTO means Data Transfer Object. It is the shape of data entering or leaving your API. Your app uses Java records for DTOs because records are simple immutable data carriers. 

public record CreateBookingRequest(
        @NotNull Long customerId,
        @NotNull Long eventId,
        @NotBlank String currency,
        @NotEmpty List<@Valid BookingItemRequest> items) {
}
  • Why: keep API data separate from database entities.
  • In your app: request DTOs enter the controller; response DTOs go back to the client.
06
Validation

Understand validation

Validation protects your app from bad input. If someone sends an empty item list or quantity zero, the app should reject the request before creating a booking.

  • @NotNull: value must exist.
  • @NotBlank: text cannot be empty or spaces.
  • @NotEmpty: list must contain at least one item.
  • @Min(1): quantity must be at least 1.
  • @DecimalMin("0.00"): price cannot be negative.

Say: "Validation happens at the API boundary, before business logic runs."

07
Service

Understand the service layer

The service layer represents use cases. A use case is a business action, such as create booking, cancel booking, list bookings, or initiate payment. 

public BookingResponse createBooking(CreateBookingRequest request) {
    request.items().forEach(item ->
        ticketingClient.reserveTickets(item.ticketTypeId(), item.quantity()));

    List<BookingItem> items = request.items().stream()
        .map(item -> toBookingItem(item, request.currency()))
        .toList();

    Booking booking = new Booking(request.customerId(), request.eventId(), request.currency(), items);
    return toResponse(bookingRepository.save(booking));
}
  • What it coordinates: ticket reservation, object creation, persistence, response mapping.
  • Why not in controller: controllers should not become messy with business workflows.
08
Domain

Understand entities and domain rules

The domain model represents the real business concepts. In this app, the important concept is Booking. A booking has a customer, event, status, reservation time, expiry, total amount, and booking items.

  • Status: new bookings start as PENDING.
  • Expiry: reservation expires after 15 minutes.
  • Total: quantity multiplied by unit price for all items.
  • Rule: confirmed bookings cannot be cancelled here.

Say: "I kept booking rules inside the Booking model so the domain behavior stays close to the data."

09
JPA

Understand database mapping

JPA maps Java objects to database tables. @Entity tells JPA that a class should be stored. @Id marks the primary key. Relationships explain how objects connect.

  • Booking: table for booking records.
  • BookingItem: table for items inside a booking.
  • @OneToMany: one booking has many items.
  • @ManyToOne: many items belong to one booking.
10
Repository

Understand repositories

A repository is the class/interface used to save and load data. With Spring Data JPA, you write an interface, and Spring creates the implementation automatically. 

public interface BookingRepository extends JpaRepository<Booking, Long> {
    List<Booking> findByCustomerId(Long customerId);
}
  • Built-in: save, findById, findAll.
  • Custom by name: findByCustomerId becomes a query automatically.
11
Transactions

Understand transactions

A transaction means a group of database operations is treated as one unit. If it succeeds, changes are saved. If an error happens, changes can be rolled back.

  • In your app: @Transactional is on BookingService.
  • Why: create/cancel/payment use cases should have consistent database behavior.
  • Read-only: methods like get booking use @Transactional(readOnly = true).
12
Errors

Understand exception handling

Instead of letting errors return random responses, your app has a global exception handler. It converts Java exceptions into clear API errors.

  • 404: booking not found.
  • 400: validation failure or invalid booking state.
  • 503: ticketing or payment service unavailable.

Say: "I use centralized exception handling so API errors are consistent."

13
Other services

Understand clients and microservices

Your booking service does not own everything. It calls other services for other responsibilities. This is a microservice idea: each service owns one business capability.

  • TicketingClient: reserves and releases ticket quantities.
  • PaymentClient: creates a payment for a booking.
  • Mock mode: lets this service run even if those other services are not running.
14
Config

Understand application.yaml

Configuration should not be hardcoded in Java. application.yaml stores settings like server port, database URL, Swagger path, and dependency service URLs.

  • Server: 8081.
  • Database: PostgreSQL connection settings.
  • Mock settings: ticketing and payment mock mode defaults to true.
15
Tests

Understand tests

Tests prove that important behavior works. Your service test checks business coordination. Your controller test checks HTTP behavior.

  • BookingServiceTest: verifies tickets are reserved and pending booking is saved.
  • BookingControllerTest: verifies valid request returns 201 and invalid request returns 400.
  • Say: "I tested both the service logic and the web/API layer."

Practice

Create Booking Flow

This is the most important flow to explain because it touches validation, external services, domain logic, and persistence.

Happy path

What happens when POST /bookings is called?

  • The controller receives a validated CreateBookingRequest.
  • The service reserves each requested ticket type through TicketingClient.
  • Request DTOs are converted into BookingItem domain objects.
  • A Booking is created with status PENDING.
  • The domain model calculates the total amount and 15 minute reservation expiry.
  • The repository saves the aggregate and the service returns a response DTO.
Failure path

What can go wrong?

  • Invalid input returns 400 through Jakarta Validation and the global exception handler.
  • A missing booking returns 404 through BookingNotFoundException.
  • A dependency failure returns 503 when a REST client call fails.
  • Invalid business states, such as paying a cancelled booking, return 400.

Database And JPA

JPA is how your Java objects become rows in PostgreSQL.

Entity mapping

Booking table

Booking is marked with @Entity, so JPA maps it to a database table. Fields like customerId, eventId, status, reservation time, expiry, and total amount become stored columns.

Relationship

One booking has many items

@OneToMany(mappedBy = "booking", cascade = CascadeType.ALL, orphanRemoval = true) means the booking owns its list of items. When the booking is saved, its items are saved too.

Repository

Spring writes common SQL for you

BookingRepository extends JpaRepository<Booking, Long>. This gives methods like save, findById, and delete behavior without writing SQL manually.

Custom query method

findByCustomerId

Spring Data understands the method name and creates a query like "find all bookings where customer_id equals this customer id." That powers GET /users/{userId}/bookings. 

public interface BookingRepository extends JpaRepository<Booking, Long> {
    List<Booking> findByCustomerId(Long customerId);
}

"I used Spring Data JPA so I could model the booking domain in Java and persist it to PostgreSQL without writing repetitive SQL for basic CRUD operations."

How The Important Parts Work Internally

This section helps you answer follow-up questions when the interviewer asks "but how does that work?"

How does Spring know about your classes?

When the app starts, @SpringBootApplication enables component scanning. Spring scans the package of the main application class and its subpackages. It finds classes with annotations like @RestController, @Service, and @Component. Then it creates objects from those classes and stores them in the Spring application context.

How does dependency injection work in your service?

BookingService has a constructor that asks for BookingRepository, TicketingClient, and PaymentClient. Spring already knows how to create those objects, so it passes them into the constructor. This is dependency injection. It means your service does not create dependencies manually, which makes the code cleaner and easier to test.

How does JSON become a Java object?

In the controller, @RequestBody CreateBookingRequest request tells Spring to read the HTTP request body. Spring uses Jackson, its JSON library, to convert JSON fields into the Java record fields. For example, JSON customerId becomes request.customerId().

How does validation work?

The controller uses @Valid. Before calling the method body, Spring checks the validation annotations inside CreateBookingRequest and nested BookingItemRequest. If validation fails, Spring throws MethodArgumentNotValidException. Your GlobalExceptionHandler catches that and returns a 400 response with validation errors.

How does JPA save a booking and its items?

Booking has a @OneToMany relationship to BookingItem with cascade = CascadeType.ALL. This means when the booking is saved, JPA also saves its items. Each item has a @ManyToOne relationship back to the booking, so the database can store which item belongs to which booking.

How does Spring Data create repository methods?

BookingRepository extends JpaRepository<Booking, Long>. Spring Data creates an implementation at runtime. For built-in methods like save and findById, Spring Data already knows what SQL to generate. For findByCustomerId, it reads the method name and generates a query based on the customerId field.

How do transactions work here?

@Transactional on BookingService means database operations inside service methods run inside a transaction. A transaction is a unit of work. If everything succeeds, changes are committed. If a runtime exception happens, database changes are rolled back. Read-only methods use @Transactional(readOnly = true) because they only load data.

How do external service clients work?

TicketingClient and PaymentClient use RestTemplate to make HTTP calls to other services. Ticketing reserves or releases ticket quantities. Payment creates a payment. In mock mode, those methods return without making real HTTP calls, which lets you demo the booking service independently.

How does error handling work?

GlobalExceptionHandler is marked with @RestControllerAdvice. It watches controller errors globally. If a booking is missing, it returns 404. If validation fails or the booking state is invalid, it returns 400. If a dependent service call fails, it returns 503. All errors are shaped as ApiError.

How does Docker Compose help?

docker-compose.yml starts both the booking service and PostgreSQL together. The app container uses database URL jdbc:postgresql://booking-db:5432/booking_db, because inside Docker Compose, services can talk to each other by service name. Your machine accesses PostgreSQL through port 5433 and the app through port 8081.

Why You Built It This Way

Interviewers often care less about the code existing and more about your reasoning.

Microservice boundary

Booking owns bookings only

Keeping inventory and payments outside this service makes the boundary cleaner. Each service can evolve around one business capability.

DTOs

API shape is separate

Request and response records prevent exposing JPA entities directly. That protects persistence details from becoming part of the public API.

Transactions

Use cases are transactional

@Transactional on the service means booking changes are handled as application-level units of work.

Validation

Reject bad data early

Jakarta Validation catches missing IDs, empty item lists, invalid quantities, and negative prices before deeper logic executes.

Mock mode

Demo independently

Ticketing and payment clients support mock mode so this service can be demonstrated before all other services are running.

Docker

Repeatable environment

The Dockerfile builds the app, and Docker Compose starts the service with PostgreSQL on the expected ports.

Why services?

To represent use cases

Creating a booking is not just saving data. It reserves tickets, creates domain objects, saves them, and returns a response. A service method is a good place to coordinate that workflow.

Why DTO records?

To keep API data clean

Java records are concise immutable data carriers. They are useful for request and response DTOs because DTOs mostly hold data and do not need complex behavior.

Why not return entities?

To protect the API contract

JPA entities contain persistence details and relationships. Returning DTOs lets you decide exactly what the API response looks like, without exposing internal database structure.

Why an enum for status?

To restrict valid states

BookingStatus can only be PENDING, CONFIRMED, CANCELLED, or EXPIRED. This is safer than using random strings.

Why a Money object?

To keep amount and currency together

Money is not just a number. The amount needs a currency. The Money value object keeps those together in booking totals and item prices.

Why BigDecimal?

To avoid floating-point money errors

Money should not use double because decimal calculations can become inaccurate. BigDecimal is the standard Java choice for precise money values.

Why Swagger?

To make the API easy to inspect

Swagger UI gives a browser page where you can see endpoints and try requests. This helps during development, demos, and interviews.

Why mock mode?

To run this service independently

In a microservice system, not every service may be ready or running. Mock mode lets Booking Service demonstrate its own behavior without needing real Ticketing and Payment services.

Why Docker?

To make running the app repeatable

Docker packages the runtime environment. Docker Compose starts both the app and database with the right ports and credentials.

Spring Security And JWT

The booking APIs are now protected with JWT bearer-token authentication.

What changed

Protected booking endpoints

Spring Security now requires a valid JWT for booking API calls. Health and Swagger endpoints stay public so the service can still be checked and documented easily.

  • SecurityConfig defines the security rules.
  • JwtProperties reads issuer and secret from configuration.
  • spring-boot-starter-security enables Spring Security.
  • spring-boot-starter-oauth2-resource-server validates bearer JWTs.
How it works

Resource server model

This service does not create users or passwords. It acts as a resource server: another auth service would issue JWTs, and Booking Service validates them before allowing access to protected endpoints. 

Authorization: Bearer <jwt-token>
Endpoint group Access Why
/actuator/health, Swagger, OpenAPI Public Useful for health checks and API documentation.
/bookings, /users/{userId}/bookings JWT required Booking data belongs to users and should not be open to anonymous callers.

"I added Spring Security and configured the service as a stateless JWT resource server. Public endpoints like health and Swagger are allowed, but booking endpoints require an Authorization bearer token. The service validates token signature and issuer using the configured JWT secret and issuer."

Interview Q and A

Practice these out loud tonight. Short, specific answers win.

Why is this a separate microservice?

Because booking is its own business capability. It has its own lifecycle, persistence, and rules. It depends on ticketing and payment, but it should not own inventory or payment processing.

What is Spring Boot doing for you?

It starts the embedded web server, scans my classes, creates Spring-managed objects, injects dependencies through constructors, loads configuration, connects to the database, and maps HTTP requests to controller methods.

What is dependency injection in your project?

BookingService needs a repository, a ticketing client, and a payment client. Instead of creating those objects manually with new, Spring passes them into the constructor. This makes the code easier to test because mocks can be injected in unit tests.

Why use DTOs instead of returning entities?

DTOs keep the API contract separate from the database model. That makes the API safer to change and prevents exposing JPA relationships or internal fields accidentally.

What is the difference between controller and service?

The controller is responsible for HTTP concerns: routes, request body, path variables, response status. The service is responsible for business use cases: reserving tickets, creating bookings, cancelling, releasing tickets, and starting payment.

How does the app calculate total price?

The Booking aggregate sums each booking item's unit price multiplied by quantity, then wraps the result in the Money value object with the selected currency.

How do you handle invalid requests?

Request records use Jakarta Validation annotations like @NotNull, @NotEmpty, @Min, and @DecimalMin. The global exception handler turns validation failures into consistent 400 responses.

What happens if the Ticketing Service or Payment Service is down?

REST client failures are caught by the global exception handler and returned as 503 Service Unavailable. In local demo mode, mock mode can be enabled so the booking service can run independently.

Why is a booking initially PENDING?

Because creating a booking reserves tickets but does not complete payment yet. The booking remains pending while the customer has time to pay before the reservation expiry.

What would you improve next?

I would add stronger compensation if reserving multiple ticket types partially fails, add integration tests with real PostgreSQL, and add an event or callback flow to confirm bookings after successful payment.

How To Demo It

Keep this sequence ready in case they ask you to run or explain the app locally.

docker compose up -d --build
curl http://localhost:8081/actuator/health
open http://localhost:8081/swagger-ui.html

# Create a booking
curl -X POST http://localhost:8081/bookings \
  -H "Content-Type: application/json" \
  -d '{"customerId":1,"eventId":10,"currency":"EUR","items":[{"ticketTypeId":5,"quantity":2,"unitPrice":25.00}]}'
Final memory hook: Booking Service receives a request, validates it, reserves tickets, creates a pending booking aggregate, stores it in PostgreSQL, and later either cancels/releases tickets or initiates payment.
